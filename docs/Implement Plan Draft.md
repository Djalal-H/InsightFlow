# InsightFlow Implementation Plan (Flexible Draft)

This roadmap is intentionally incremental. The immediate priority is a clean repository, a small
dependency surface, hosted model APIs, and Qdrant. Later stages can change as implementation and
evaluation expose better choices.

## Stage 1: Repository and development foundation

**Goal:** Establish a maintainable Python project without prematurely building the product.

### Deliverables

- Modular `src/insightflow` package with API, agent, RAG, provider, storage, and core boundaries.
- FastAPI application factory plus liveness and readiness endpoints.
- Provider-independent chat and embedding contracts implemented through LiteLLM.
- Qdrant adapter that only accepts vectors generated by hosted embedding APIs.
- Environment configuration through Pydantic Settings and `.env.example`.
- Pinned runtime and development requirements.
- A dedicated Python virtual environment.
- Docker Compose containing only Qdrant with persistent storage and a health check.
- README instructions and updated architecture documentation.

### Explicit deferrals

- No package installation or model download during the initial scaffold.
- No local LLM, embedding model, reranker, GPU runtime, or inference service.
- No document parser until the supported ingestion formats are selected.
- No application Dockerfile, LiteLLM Proxy, Redis, MCP, auth, observability stack, or CI/CD yet.

## Stage 2: Minimal hosted-model workflow

**Goal:** Prove model-provider configuration and LangGraph execution before adding retrieval.

### Core work

- Add a request/response API schema and a non-streaming query endpoint.
- Implement one LangGraph node that calls the chat-provider contract.
- Validate provider configuration without hard-coding a vendor.
- Map provider errors, timeouts, and rate limits into stable application errors.
- Mock LiteLLM calls in unit tests and keep real API calls opt-in.

### Acceptance criteria

- One configured hosted model can answer a request through the API.
- Changing to another LiteLLM-supported provider requires environment changes only.
- Missing credentials or invalid models fail with a clear, sanitized error.

## Stage 3: Basic ingestion and RAG

**Goal:** Deliver an end-to-end grounded-answer workflow using Qdrant.

### Core work

- Select the first document formats; begin with plain text and add PDF only if required.
- Normalize documents, split them into traceable chunks, and preserve source metadata.
- Generate dense embeddings through the LiteLLM embedding adapter.
- Create and migrate a Qdrant collection whose vector size matches the configured model.
- Store one chunk per Qdrant point and implement dense similarity retrieval.
- Build a LangGraph path: query embedding → retrieval → grounded generation → structured answer.

### Acceptance criteria

- Ingested content is searchable after restart.
- Answers include source identifiers for their retrieved context.
- Tests verify vector/payload mapping without contacting a hosted model by default.

## Stage 4: Workflow expansion and state persistence

**Goal:** Add agentic behavior only where it improves the RAG workflow.

### Core work

- Add explicit planner/router, retriever, answer, and verifier nodes.
- Introduce conditional edges and bounded retry loops.
- Define structured final output containing answer, sources, and execution metadata.
- Add a persistent LangGraph checkpointer after resumability requirements are defined.
- Evaluate whether conversational memory belongs in checkpoints, Qdrant, or both.

### Optional work

- Streaming through Server-Sent Events.
- Parallel retrieval for decomposed queries.
- Human approval for sensitive future tools.

## Stage 5: MCP tool integration

**Goal:** Add safe external actions through a standard protocol.

### Core work

- Add the official MCP SDK only when this stage starts.
- Implement an MCP client boundary used by LangGraph nodes.
- Integrate one bounded, low-risk tool before adding a registry.
- Enforce input schemas, timeouts, error handling, output-size limits, and audit metadata.
- Test tool/no-tool routing and graph termination.

### Optional work

- Tool discovery and dynamic registration.
- Auth propagation and result caching.
- Additional MCP servers for search, repositories, or databases.

## Stage 6: Retrieval quality and memory

**Goal:** Improve measured retrieval failures rather than adding techniques by default.

Candidate improvements include query rewriting, metadata filters, hybrid dense/sparse retrieval,
parent-child context, API-backed reranking, context compression, corrective retrieval, multi-hop
queries, and retrieval-based conversation memory. Each addition should be compared on a small labeled
evaluation set before becoming part of the default path.

Local SentenceTransformers, FastEmbed, cross-encoders, or other downloaded models remain outside the
default architecture. Reranking and summarization should use hosted APIs unless this policy is
explicitly revisited.

## Stage 7: Observability, evaluation, and reliability

**Goal:** Make execution quality and operational failures visible.

### Core work

- Correlate logs across API requests, graph nodes, provider calls, Qdrant queries, and tools.
- Record latency, token use, retrieval counts, retries, and error categories without logging secrets.
- Add deterministic regression datasets for retrieval and structured-output behavior.
- Add API-backed evaluation only as an opt-in, budgeted job.

### Technology candidates

- OpenTelemetry for instrumentation
- Langfuse for LLM traces and prompt management
- Prometheus and Grafana if service-level metrics require them

These systems should not all be adopted automatically.

## Stage 8: Security and production readiness

**Goal:** Prepare a proven workflow for controlled deployment.

### Core work

- Authentication, authorization, rate limiting, and quotas.
- Secret management and removal of sensitive prompt/response fields from logs.
- Prompt-injection controls and explicit authorization for consequential tools.
- Application Dockerfile and CI checks.
- Backup and migration procedures for Qdrant and graph checkpoints.
- Load tests and provider-failure simulations.

Kubernetes, multiple application services, Redis, a LiteLLM Proxy, and infrastructure as code should
be selected only if deployment requirements justify them. Dockerizing Qdrant locally does not require
containerizing the application at Stage 1.

## Dependency policy

- `requirements.txt` contains direct runtime dependencies only.
- `requirements-dev.txt` includes runtime requirements plus test, lint, formatting, and typing tools.
- Direct dependencies are pinned and updated deliberately after compatibility checks.
- Provider-specific packages, document loaders, and infrastructure clients are added in the stage
  that first imports them.
- Avoid umbrella frameworks when a smaller official client or focused package meets the need.
- Never add local-model packages implicitly through optional extras.

## Current success criteria

The project foundation is ready when the structure is understandable, the virtual environment is
isolated, dependencies are recorded but not unnecessarily broad, model and Qdrant settings are
documented, and no active design requires local inference. Functional RAG quality targets begin only
after the Stage 3 workflow exists.
